Foodpanda & GrabFood Integration Requirements

Complete requirements checklist for integrating both delivery platforms with the Dr. Humba Restaurant Inventory Management System.

Prepared: June 16, 2026

1. Foodpanda (Delivery Hero Middleware) Requirements

ℹ️ Note: We already have the official Delivery Hero Integration Middleware API specification file (middlewareExternalApi.yaml). This is the same system that powers Foodpanda globally.

1.1 Business Prerequisites

Requirement Details Status
Registered Foodpanda Merchant Dr. Humba must be an active Foodpanda merchant-partner with a signed commercial agreement ☐ Verify
Chain ID Foodpanda assigns a unique chainCode to your restaurant chain. Contact your Foodpanda Account Manager ☐ Request
POS Vendor IDs Each Dr. Humba branch maps to a posVendorId on Foodpanda's side. One per branch ☐ Request per branch
API Credentials client_id and client_secret for OAuth 2.0 authentication. Issued via the Delivery Hero Partner Portal or Account Manager ☐ Request
Integration Type Agreement Confirm whether doing a direct integration (accept/reject orders via API) or indirect (accepted on Foodpanda device first, then forwarded) ☐ Clarify

1.2 Technical Prerequisites

Requirement Details
Publicly accessible HTTPS endpoint Foodpanda dispatches orders to your POS plugin via webhooks. Your server must expose endpoints that Foodpanda can reach over the internet
OAuth 2.0 JWT Authentication All API calls to the Middleware require a Bearer token obtained from POST /v2/login with grant_type: client_credentials
Webhook handling You must implement the POS Plugin API endpoints (the "callback" side) to receive order dispatches and status updates from Foodpanda
JSON catalog format Menu imports use the new Catalog Import API (replaces legacy XML). Full catalog pushes only — partial updates not yet supported
Image hosting Menu item images must be served from publicly accessible, immutable URLs (max 20MB, max 16 megapixels)

1.3 API Endpoints — Your System Calls Middleware

These are the endpoints your system calls on the Delivery Hero middleware:

Endpoint Method Purpose Priority
/v2/login POST Obtain JWT authentication token Required
/v2/order/status/{orderToken} POST Accept, reject, or mark order as picked up Required
/v2/orders/{orderToken}/preparation-completed POST Notify rider that food is ready for pickup Recommended
/v2/orders/{orderToken}/adjust-preparation-time POST Dynamically adjust kitchen prep time Optional
/v2/order/{orderToken}/modifications/product POST Modify order products (out-of-stock substitutions) Optional
/v2/chains/{chainCode}/remoteVendors/{posVendorId}/availability GET / PUT Check or update store open/closed status Recommended
/v2/chains/{chainCode}/catalog PUT Push full menu catalog to Foodpanda Recommended
/v2/chains/{chainCode}/vendors/{posVendorId}/items/availability PUT Toggle item availability (86'd items) Recommended

1.4 Webhook Endpoints — Foodpanda Calls Your System

These are the POS Plugin endpoints your system must implement for Foodpanda to call:

Endpoint You Host Purpose Priority
POST /dispatch-order Receives incoming order payload (items, prices, customer info, delivery details, callback URLs) Required
POST /update-order-status Receives order status updates (cancelled, rider assigned, product modification result) Required
POST /catalog-import-status Receives catalog import progress callbacks (success/failure of menu pushes) Optional

1.5 Order Lifecycle Flow (Foodpanda)

  1. Customer places order on Foodpanda app
  2. Delivery Hero Middleware dispatches order to our system via webhook (POST /dispatch-order)
  3. Our system validates ingredient stock availability
  4. Our system calls POST /v2/order/status/{token} with order_accepted
  5. Our system deducts ingredients from branch inventory and prints kitchen ticket
  6. Kitchen completes preparation → our system calls POST /v2/orders/{token}/preparation-completed
  7. Middleware notifies rider that food is ready
  8. Rider picks up order → Middleware sends POST /update-order-status (picked_up) to our system
  9. Sale recorded in Sales History with sale_category = 'Foodpanda'

2.5 Order Lifecycle Flow (GrabFood)

  1. Customer places order on GrabFood app
  2. Grab sends new order webhook to our system
  3. Our system validates ingredient stock availability
  4. Our system calls POST /partner/v1/order/prepare to accept
  5. Our system deducts ingredients from branch inventory and prints kitchen ticket
  6. Kitchen completes preparation → our system calls POST /partner/v1/orders/mark (ready)
  7. Grab notifies rider for pickup
  8. Grab sends pickup webhook to our system
  9. Sale recorded in Sales History with sale_category = 'Grab'

3. Data Mapping to Dr. Humba System

How platform data maps to the existing Supabase database schema:

Platform Field Dr. Humba Field Database Table Notes
Order items sale_items sale_items Map platform remoteId → menu_item_id
Order total total_amount sales Platform-calculated total
Order token / ID reference_number sales External platform order reference
Platform name sale_category sales Set to 'Foodpanda' or 'Grab' automatically
Payment method payment_method sales Always 'other' (platform handles payment)
Branch mapping branch_id sales Map posVendorId / merchantID → branch UUID
Cashier cashier_id sales Use a system service account per platform

New Database Tables Needed

Table Name Purpose
platform_credentials Store chainCode, vendorIds, API keys per branch per platform
platform_orders Track order lifecycle states (received → accepted → preparing → ready → picked up → completed)
platform_menu_mappings Map Dr. Humba menu_item_id to platform-specific product IDs

4. Infrastructure Requirements

4.1 Webhook Server

⚠️ Important: The current system is a client-side React app with Supabase backend. Webhooks require a server-side component that can receive HTTP requests from Foodpanda/Grab.
Option Approach Pros Cons
Supabase Edge Functions (Recommended) Deploy Deno-based webhook handlers on Supabase No additional infra, same database access Cold starts, 150s timeout limit
Vercel/Netlify Serverless Deploy API routes as serverless functions Easy deployment, good scaling Separate from database, need Supabase client
Dedicated Backend Express.js / Fastify on a VPS (e.g., DigitalOcean) Full control, persistent connections More ops overhead, separate deployment

4.2 Security Requirements

Requirement Details
HTTPS / TLS All webhook endpoints must be served over HTTPS
Webhook signature validation Verify that incoming requests are genuinely from Foodpanda/Grab (via HMAC signatures or IP allowlisting)
Credential encryption Store API keys encrypted at rest in the platform_credentials table
Rate limiting Implement rate limiting on webhook endpoints to prevent abuse
Idempotency Handle duplicate webhook deliveries gracefully (use order token as idempotency key)

5. Onboarding Checklist

Phase 1: Business Setup (Week 1–2)

  • Confirm active Foodpanda merchant status for all branches
  • Confirm active GrabFood merchant status for all branches
  • Contact Foodpanda Account Manager → request chainCode, posVendorId per branch, API credentials
  • Contact Grab Business Development → request merchantID per branch, developer portal access
  • Sign integration agreements with both platforms
  • Receive sandbox/staging credentials from both platforms

Phase 2: Technical Setup (Week 3–4)

  • Set up Supabase Edge Functions (or chosen webhook server)
  • Implement OAuth 2.0 token management for both platforms
  • Create platform_credentials, platform_orders, platform_menu_mappings database tables
  • Build webhook receiver: POST /webhook/foodpanda/order
  • Build webhook receiver: POST /webhook/grab/order
  • Map each branch's posVendorId / merchantID to Supabase branch_id

Phase 3: Order Flow (Week 5–7)

  • Implement order acceptance flow (validate stock → accept → deduct ingredients → kitchen ticket)
  • Implement order rejection flow (insufficient stock → reject with reason)
  • Implement order cancellation handling (restore ingredients via existing void/refund logic)
  • Build real-time order dashboard in the Dr. Humba UI
  • Add platform order sound notifications / alerts

Phase 4: Menu Sync (Week 7–8)

  • Build menu export: Dr. Humba menu_items → Foodpanda catalog JSON format
  • Build menu export: Dr. Humba menu_items → GrabFood menu format
  • Implement auto-86 (mark unavailable) when ingredient stock hits low-stock threshold
  • Test full catalog push to both platforms in staging environment

Phase 5: Testing & Go-Live (Week 9–10)

  • End-to-end testing in both platform sandboxes
  • Test order → ingredient deduction → Z-Report reconciliation
  • Test store pause/unpause from Dr. Humba dashboard
  • Pilot launch with one branch
  • Monitor for 1 week, fix edge cases
  • Roll out to all branches
