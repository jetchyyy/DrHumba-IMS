Your Current System Setup

The RESTOChain system is designed to prioritize transaction safety and data integrity by utilizing a zero-client-logic database architecture.

Vite + React Frontend
A fast, interactive browser-based panel where branch managers view reports and cashiers record in-store POS checkouts.
Supabase Database
Houses raw tables, user roles (RBAC), and transaction logs in a cloud-hosted environment.
Secure Server Functions (RPC)
100% of business logic (like checking recipe ingredients and updating stock balances) is performed securely inside the database, preventing race conditions or inventory bugs.

Safe & Relational

Currently, when a cashier completes a sale inside the physical restaurant, the browser calls a secure DB function:

fn_process_sale(
  p_branch_id => 'Branch A',
  p_items => [Classic Burger, Onion Rings]
);

If ingredients are available, stock is deducted and the sale is recorded instantly. If not, it rolls back to prevent issues.

The Challenge: Webhooks

To connect Foodpanda & Grab, the delivery companies need to send order notifications (webhooks) to our system. But they cannot talk directly to a browser app.

The Limitation (Browser App Only)

A React browser app is only active when a cashier has it open. If the browser tab is closed or a computer goes to sleep, incoming orders from Foodpanda or Grab have nowhere to go and will fail.

The Solution: A Webhook Server

We need a lightweight backend server. Think of it as a Digital Receptionist. It is a simple server that sits in the cloud, stays awake 24/7, listens for incoming orders from Grab/Foodpanda, and immediately processes them.

How the "Receptionist" Works:

  • 1. Listens 24/7: Stands ready to receive customer orders from Foodpanda/Grab webhooks at any second of the day.
  • 2. Verifies Stock: Instantly connects to the Supabase database, calling our RPC function to check if the kitchen has enough raw materials for the order.
  • 3. Confirms/Deducts: If ingredients are available, it tells the delivery platform "Order Accepted!", deducts inventory, and sends the ticket to the kitchen.

Required API Credentials & Handshakes

To allow secure communication, the client must request and provide specific credentials from Foodpanda and Grab.

Chain Code
Unique chain identifier assigned by Foodpanda to map all of Dr. Humba's branches.
POS Vendor IDs
Individual branch IDs. One unique code generated by Foodpanda for each location (e.g., Branch A, Branch B).
OAuth Credentials
Client ID and Client Secret keys generated via the Delivery Hero Partner Portal to securely send catalog syncs and order status updates.
Merchant ID
Unique merchant identification strings representing each branch store registered on the GrabMerchant app.
Partner API Keys
OAuth 2.0 credentials (client keys) requested from the Grab Developer Portal to allow integration communication.
Sandbox Access
Test keys and test store access required so we can run orders safely in staging before going live.

Where will the new server run?

We have two primary, budget-friendly options to deploy this server, both ensuring maximum uptime and security.

Recommended
Supabase Edge Functions

Deploying small, serverless scripts (handlers) directly within your existing Supabase project database environment.

Pros
  • Zero hosting cost overlap
  • Direct connection to database
  • Instant automatic scaling
Cons
  • 150-second run timeout
  • Initial delay on cold boot
Dedicated Node.js / Express Server

A small dedicated server hosted on platforms like Render or DigitalOcean VPS, running constantly.

Pros
  • Always warm (0ms delay)
  • Infinite control over tasks
  • Allows persistent sockets
Cons
  • Small monthly cost ($5–$10)
  • Requires separate account
